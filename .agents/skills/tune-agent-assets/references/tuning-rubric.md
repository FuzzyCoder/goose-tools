## Progressive Disclosure (PD)

### Line and Token Thresholds

| Asset | Line Target | Body Token Target | Action when exceeded |
|---|---|---|---|
| SKILL.md | ≤200 lines | ≤5,000 tokens | Extract dense material to `references/` |
| SKILL.md (hard ceiling) | 500 lines | — | Must extract; file must shrink below 200 |
| Root AGENTS.md | ≤100 lines | — | Trim; move detail to skill references |
| Subdirectory AGENTS.md | ≤80 lines | — | Trim; move detail to skill references |

### Per-Skill Description Ceiling

`len(description) > 500` is the threshold for "oversized" (applies to the YAML-parsed
description string; excluding the YAML key, indentation, and surrounding quotes).
When trimming:
- Every quoted trigger phrase must appear verbatim in the trimmed description.
- Every `Not for ...`, `Do NOT use for ...`, or `NOT for ...` clause must appear verbatim.
- No trigger phrase or boundary may be removed, reworded, or paraphrased.
- Descriptive prose (non-trigger, non-boundary text) may be shortened.

### Listing-Cap Budget

`16,000 chars` total across all tracked SKILL.md descriptions.
- "Listing total" = sum of `len(name) + len(description)` across all tracked SKILL.md files.
- "Average chars per skill" = listing total / number of tracked SKILL.md files.
- See `docs/progressive-disclosure-compliance.md` for measured baseline and post-trim totals.

### Extraction Decision Criteria

Extract to `references/<topic>.md` when the section meets ANY of these conditions:
1. The section is a lookup table or reference chart rarely consulted at runtime
2. The section contains extended examples that exceed 20 lines
3. The section contains procedural detail not needed to decide whether to invoke the skill
4. Removing the section from the main file would bring the SKILL.md under 200 lines
5. The section's content is stable and unlikely to need inline visibility for quick orientation

Keep inline when:
- The content is critical context for every invocation (e.g., `## Autonomy Gates`, `## When to Use`)
- The section is short (≤10 lines) and extraction would create overhead larger than the benefit
- Extraction would break the skill's linear execution flow

### Extraction Pattern

1. Create `references/<topic>.md` with a top-level heading and table of contents
2. In the SKILL.md body, replace the dense section with a 1–2 line summary and a link:
   ```markdown
   [references/<topic>.md](./references/<topic>.md)
   ```
3. Add the file to `## References Structure` at the bottom of the SKILL.md

### Link Pattern

Always use relative links within the same skill directory:
```markdown
[references/tuning-rubric.md](./references/tuning-rubric.md)
```

Never use absolute paths or repo-rooted paths.

## Agent Harnessing (AH)

### When to Add `## Autonomy Gates`

Add `## Autonomy Gates` immediately after `## When to Use` in skills that:
- Edit files (any file write, create, or delete)
- Commit to a git repository
- Push to a remote
- Post comments on PRs or issues

Do NOT add `## Autonomy Gates` to:
- Review-only skills (produce reports only, never edit)
- Reference-only skills (explain conventions, no side effects)
- Router/dispatcher skills that only invoke sub-agents without direct file writes

### Gate Table Format

Use a Markdown table with two columns for skills with many distinct gated actions:

```markdown
## Autonomy Gates

| Action | Authorization |
|---|---|
| Edit `.agents/skills/<skill>/SKILL.md` | Authorized by trigger phrase |
| Commit to branch | Authorized by trigger phrase |
| Push to remote | Authorized by trigger phrase |
```

Use the narrative format for skills with a single gated action type:

```markdown
## Autonomy Gates

Commit and push are **only authorized** when the user invokes this skill via one of its
explicit trigger phrases.
```

### Canonical Examples

- `fix-issue/SKILL.md` lines 17-32 — full table format with multiple action rows
- `update-deps/SKILL.md` lines 28-34 — narrative format for a single gate
- `review-and-fix-pr/SKILL.md` lines 36-53 — table format with NOT authorized actions

### Standing NOT Authorized List

No skill's `## Autonomy Gates` should authorize edits to:
- `CAPABILITIES.md` — user-managed catalog; hand-edits do not persist across regeneration
- `TODO.md` — auto-generated by `bin/goose-tools todo`; hand-edits do not persist
- `INVENTORY.md` — auto-generated by `bin/goose-tools inventory`; hand-edits do not persist

## Token Reduction (TR)

### Priority Order (Highest Ratio / Lowest Risk First)

**1. Remove preamble sentences** that restate the section heading.
- Example: under `## When to Use`, remove "This section describes when to use this skill."
- Risk: None. Impact: 2–5 lines per section.

**2. Collapse prose to bullets** — multi-sentence paragraphs explaining a list of items
can almost always be a bullet list at half the token count.
- Risk: Low (readability preserved). Impact: 10–30% body token reduction.

**3. Extract large examples** — command blocks >20 lines and extended examples belong in
`references/` where they are available but not loaded on every invocation.
- Risk: Low (content preserved). Impact: 20–50% reduction when applicable.

**4. Deduplicate command blocks** — if the same command appears in multiple steps, extract
it to a single canonical location and reference it.
- Risk: Low. Impact: variable.

**5. Remove padding phrases** — hedge phrases like "it is important to note that",
"please keep in mind", or "as mentioned above" add tokens without meaning.
- Risk: None. Impact: 1–3% reduction.

**6. Shorten frontmatter description** — descriptions that exceed 6 lines often repeat
`## When to Use` bullets. Trim to the shortest description that preserves trigger coverage.
- Risk: Medium (may reduce triggering accuracy). Only trim if obvious redundancy exists.

### Reduction Targets

| Condition | Target |
|---|---|
| SKILL.md body tokens > 5,000 | Reduce to ≤5,000 |
| SKILL.md lines > 200 | Reduce to ≤200 or extract |
| Section >40 lines and optional at runtime | Extraction candidate |

### What NOT to Reduce

- `## Autonomy Gates` — must remain fully inline; never extract
- `## When to Use` bullets — these are triggering signals; keep all
- `## Validation` checklist — agents need this inline during execution
- Error messages and halt conditions — must be inline for fast-path reads
